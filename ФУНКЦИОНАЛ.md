# BOBO — описание функционала

Платформа ежедневной доставки продукции пекарни в кафе сети **BOBO**.
Интерфейс приложения — на английском; это описание — на русском.

---

# Часть 1. Для заказчика — что система делает

## Роли и доступ
Все учётные записи (логин + пароль) заводит **Администратор**. Саморегистрации нет.

| Роль | Кто это | Что делает |
|---|---|---|
| **Администратор** | владелец системы | заводит участников, ведёт каталог и цены, настраивает делегирование |
| **Пекарня** | производство | формирует недельные заказы, отправляет курьером, разбирает инциденты, выставляет счета |
| **Кафе** (несколько) | точки сети | корректируют и подтверждают заказ, принимают доставки, сообщают об инцидентах, видят счета |
| **Курьер** | развоз | принимает партию у пекарни, подтверждает доставку фото, может сообщить об инциденте |

## Жизненный цикл

**1. Каталог и цены.** Админ ведёт перечень продуктов (категория, единица «за штуку/за кг», цена, флаг «можно примечание», архив) и цены. Может делегировать пекарне право править каталог и/или цены.

**2. Недельный заказ — два типа подтверждений.**
- **Тип 1 — план недели (будущая неделя).** Пекарня предлагает перечень на неделю → кафе либо **подтверждает** как есть, либо **вносит правки и отправляет** → пекарня **принимает или отклоняет** правки (отклонение = откат к предложению пекарни). У кафе есть срок — **36 часов до начала недели** (настраивается); если не успели — заказ авто‑подтверждается по предложению пекарни. На текущей (уже идущей) неделе этот сценарий не работает.
- **Тип 2 — правки по дням (подтверждённая неделя).** Кафе запрашивает изменение количества по конкретному дню — но **только до «катоффа» этого продукта** (крайнего срока). Пекарня принимает или отклоняет каждую заявку.

**3. Доставка.** Пекарня назначает курьера на день (**Dispatch**) — заказ снимается «снимком» в доставку. Курьер подтверждает **приёмку партии** у пекарни, затем по каждому кафе грузит **фото** и отмечает «доставлено». Кафе получает **тикет о доставке** с фото.

**4. Инциденты.** Курьер **или** кафе сообщает о порче/потере продукта на доставке (с опциональным фото). Пекарня **подтверждает или отклоняет**.

**5. Счета (инвойсы).** Раз в неделю пекарня запускает **пакетную генерацию по всем кафе** за завершившуюся неделю. Доступно, только если:
- неделя **закончилась**, и
- по этой неделе **нет неразобранных инцидентов**.

Считаются **доставленные** позиции по ценам на момент заказа. **Подтверждённые инциденты исключаются из суммы**, но показываются отдельной строкой. Кафе без доставок за неделю **пропускаются** (пустой счёт не создаётся). Затем — отправка счетов кафе.

## Ключевые правила
- **Катофф по продукту** — крайний срок правок по конкретному дню доставки (например, за 18 ч).
- **Дедлайн недели** — 36 ч до начала недели (настраивается глобально).
- **Все кафе** биллятся **еженедельно**.
- **Цена в счёте** — снимок на момент заказа (последующее изменение цены в каталоге на старые счета не влияет).
- Валюта — **GBP**, без НДС (в прототипе).

# Часть 2. Для Марии — экраны, код и запуск

## Технологии
**Next.js (App Router) + React + Tailwind CSS.** Дизайн задаётся **классами Tailwind прямо в компонентах** (`.tsx`). Фирменный цвет — `bobo` (тёплый коричневый) в `tailwind.config.ts` (`bobo`, `bobo-light`, `bobo-dark`).

## Как запустить локально
Требуется Node 20+ и npm.
```bash
npm install
npm run db:push     # создаёт локальную БД (SQLite, файл prisma/dev.db)
npm run db:seed     # демо-данные (каталог, кафе, пользователи)
npm run dev         # http://localhost:3000
```
Сбросить БД к демо-состоянию: `npm run db:reset`.

## Важно (чтобы не сломать)
- **Не коммитить** `node_modules`, `.env`, `prisma/dev.db`, `public/uploads` — они уже в `.gitignore`.
- Для дизайна меняем разметку и Tailwind-классы в компонентах; пропсы/данные приходят сверху — структуру данных лучше не менять.
- После правок: `npm run dev` и проверить в браузере под нужной ролью.
